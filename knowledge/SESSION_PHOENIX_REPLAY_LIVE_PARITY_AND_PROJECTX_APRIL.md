## 1. Background questions (conversation arc)

### 1.1 Live realism vs “regular” backtest

Earlier work compared **live-pace / realism replay** (causal `run_scan_once` stepping, often `touch_strict` + slippage) to a **contiguous traditional** `run_backtest` over the same date range. Example artifact:

- `reports/live_realism/balanced_50k_survival/2025_04_28_to_2026_04_28/LIVE_REALISM_VS_TRAD_BACKTEST.md`

That report is generated by `scripts/generate_live_replay_vs_backtest_report.py` (see footer in the MD file).

### 1.2 Realistic replay vs Project X April fills

The user wanted to compare **realism replay trades** to **actual Project X / Tradovate exports** (`REAL_PROJECT_X_APRIL/trades_export.csv`).

**Deliverable added:** `scripts/compare_realism_replay_vs_projectx_fills.py`

- Loads broker CSV + `phoenix_live_replay_trades.csv`.
- Normalizes Project X `ContractName` to engine roots (**critical:** `MNQM6` → `MNQ`, not `MNQM`; use longest-first roots `MGC`, `MNQ`, `YM`, `CL`, etc. in `_INSTRUMENT_ROOTS`).
- Compares aggregates over the **broker export’s calendar span** and optional greedy 1:1 matching (instrument, direction, size, entry ET within `--entry-match-seconds`, default 180; often use 300).
- By default drops replay rows whose `exit_reason` contains `close_all_flatten` for matching (configurable).
- Writes under `reports/benchmarks/`:
  - `realism_replay_vs_projectx_april.md`
  - `realism_replay_vs_projectx_april_summary.json`
  - `realism_replay_vs_projectx_april_matched_pairs.csv` (when any matches exist)

**Example run** (after generating replay trades for `Balanced_50k_survival` on the same window as benchmark data):

```bash
python3 scripts/run_phoenix_realism_harness.py \
  --data-dir Data-DataBento/april_frontmonth \
  --start-date 2026-03-28 --end-date 2026-04-28 \
  --preset Balanced_50k_survival --no-sleep

python3 scripts/compare_realism_replay_vs_projectx_fills.py \
  --replay-trades-csv reports/live_realism/balanced_50k_survival/2026_03_28_to_2026_04_28/phoenix_live_replay_trades.csv \
  --entry-match-seconds 300
```

**Interpretation:** Portfolio totals and per-trade PnL rarely match exactly (bar timestamps vs tick timestamps, fees, `close_all_flatten` legs, fill mode). The script quantifies **aggregate** differences and **time-aligned** subset matches.

### 1.3 “Is live wrong if numbers differ?”

**Architecture fact:** Live Phoenix uses the **same** `run_backtest` / `fast_engine` for **signals**: `fresh_entries_for_latest_bar` in `projectx/strategy/phoenix_auto.py` diffs full bars vs `bars.iloc[:-1]`.

**Divergence sources (not a second secret strategy):**

- `execution.phoenix_entry_order: limit` → resting **stop** at engine `entry_price`; `entry_breakout_stop_valid` can **skip** sends when last price has already traded through (`projectx/main.py`).
- Gateway **partial bars**, API **tickValue** rounding vs static `TICK_VALUES`, **risk / dedupe / arm** maintenance, **data path** (Gateway vs DataBento).

Canonical doc: `docs/PHOENIX_LIVE_TRADING_AND_BACKTEST_PARITY.md`

## 2. Replay accuracy: what `phoenix_live_pace_replay` does **not** do

`scripts/phoenix_live_pace_replay.py` calls **`run_scan_once`** (same as `projectx.main` scan path for **hits**) and optionally aggregates closed trades via **`run_backtest`** on the loaded bar window.

**It does *not* simulate:**

- ProjectX **executor** or real orders
- **Arm** buy-stop / sell-stop placement (`phoenix_arm_orders`)
- **`entry_breakout_stop_valid`** (until the change in §3)
- Risk manager / dedupe persistence / partial-bar Gateway feed

So replay ≠ full live execution stack; it is the closest **local** analogue for **engine + polling** on OHLC.

## 3. Work completed: live **stop @ trigger** parity in replay

To narrow the gap for “would live send this hit on a limit/stop entry path?”, the following was added.

### 3.1 Module: `projectx/strategy/phoenix_replay_parity.py`

- `live_stop_entry_eligibility(...)` — mirrors `projectx/main.py` logic: if `phoenix_limit_entry` (stop at trigger), require valid `entry_price` and `entry_breakout_stop_valid(side, trigger, last_close, tick_size)`; market mode skips the check.
- `filter_hits_with_live_stop_gate(...)` — filters `run_scan_once` hit tuples and returns skip counts by reason (`entry_breakout_stop_invalid`, `invalid_entry_price`, `no_bars`, etc.).

### 3.2 `scripts/phoenix_live_pace_replay.py` changes

New flags:

| Flag | Meaning |
|------|--------|
| `--live-entry-parity` | After each `run_scan_once`, remove hits live would skip for stop@trigger vs **last bar close** (same idea as `main.py` before `execute_dollar_risk_bracket`). |
| `--parity-market-entry` | Mimics `--phoenix-market-entry`: do **not** apply stop-validity. |

**Logging:** lines show `raw_hits=` and `eligible_hits=` when parity is enabled.

**Stats JSON:** `total_hit_events` remains **raw engine** hit count; `live_entry_parity` block includes `total_eligible_hit_events`, `stop_gate_skip_reasons`, `phoenix_limit_entry`, etc. Fingerprints / `NEW=` lines use **eligible** hits when parity is on.

**Still not covered:** arms, risk, dedupe store behavior, Gateway partial bars.

### 3.3 VPS helper: `scripts/run_phoenix_vps_parity_suite.py`

Wraps `phoenix_live_pace_replay.py` with `--step-mode bar`, `--no-sleep`, `--live-entry-parity`, configurable `--bars-window`, fill mode, slippage, optional `--max-steps`.

- **Windows UTF-8:** sets `PYTHONIOENCODING=utf-8` and `PYTHONUTF8=1` on win32 for subprocess hygiene.
- Default output: `reports/vps_parity_<start>_to_<end>.json` (override `--out-json`).

Full PowerShell / bash examples are in the script docstring.
